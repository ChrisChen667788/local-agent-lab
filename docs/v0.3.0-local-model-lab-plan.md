# v0.3.0 Local Model Lab Plan

Last updated: 2026-04-08

## Why this plan exists

这份计划专门回答四件事：

1. LM Studio 截图里那些细粒度推理控制项，哪些值得加入 Local Agent Lab。
2. 之前已经确认“最值得移植”的四个能力，如何落成产品方案、页面信息架构和技术实现清单。
3. 如何在 Web 端增加“定时从内存卸载本地模型”的能力，缓解长期占用内存。
4. 如果后续支持本地模型微调，第一版最适合做哪些功能，哪些不要一开始就做。

这份计划默认基线是：

- `v0.2.x` 已经把本地 runtime、benchmark、mixed compare、progress recovery 收稳。
- `v0.3.0` 的目标是**在原项目里增强本地模型实验能力**，不是另起一个新项目。

## Scope guardrails

这份计划必须遵守下面这些边界：

- 保留当前 `Next.js + app router + Agent/Admin shell` 框架，不另起技术栈。
- 保留 `/agent` 和 `/admin` 作为主工作面；新能力优先以内嵌模式、抽屉、子区块、实验路由的方式进入。
- 保留当前项目的核心主线：`agent workflow + benchmark ops + runtime ops`。
- Compare、presets、prompt template、structured output 这些能力，属于现有工作台的增强层，不是新的独立产品。
- 微调能力在第一阶段只允许作为实验能力接入现有工作台，不允许把项目重心改成训练平台。
- UI 必须延续当前 `/agent` 和 `/admin` 的信息架构与视觉语言，不做完全重构式的新产品界面。

## A0. 从 Google AI Studio 和 LM Studio 反推，哪些能力适合移植

这一步不做“谁抄谁”的功能清单，而是看两类成熟产品各自证明了什么价值，再判断哪些东西长到 First LLM Studio 里不会违和。

### Google AI Studio 当前更强的卖点

基于官方文档，Google AI Studio 现在最突出的点有：

- 快速试 prompt，并直接 `Get code` 进入 Gemini API 代码接入
- `Run settings` 面板里把参数、safety、structured output、function calling、code execution、grounding 放在同一个实验入口
- `Prompt gallery` 提供现成案例和可复用 prompt ideas
- `Build mode` 可以从文字描述生成可运行原型，并带代码与 live preview
- AI Studio 本身还承担 key 管理、usage 观察和原型验证入口

这些能力的中心思想不是“参数多”，而是：

- 从实验到代码的路径非常短
- 模型能力是按“能力模块”来理解的，而不是按一大堆零散控件来理解
- starter examples / gallery 能明显降低上手门槛

### LM Studio 当前更强的卖点

基于官方文档，LM Studio 当前最值得参考的点有：

- per-model defaults
- presets 的保存、导入、分享和 URL 分发
- prompt template override
- MCP host / local+remote MCP server 接入
- 本地 API server 的 JIT load / auto unload / only keep last loaded model
- server settings：本地网络暴露、认证、CORS、MCP 安全开关
- `LM Link` 这种“把本地 AI 工作负载路由到别的机器”的产品方向

这些能力的中心思想是：

- 本地模型生命周期管理做得很成熟
- 配置复用与分享门槛低
- 本地工作负载可以逐步从“单机 GUI”进化成“本地 infra”

### 哪些东西不适合直接搬

#### 1. Google AI Studio 式的完整 app prototyping / vibe coding

不建议直接移植。

原因：

- 你的项目主线是 `local-first + compare + benchmark + runtime ops`
- 如果硬塞一个“生成完整 Web app”的 Build mode，产品会变得很散
- 这会和 Compare / Benchmark / Runtime 的核心价值抢注意力

#### 2. LM Studio 式的通用模型下载市场首页

也不建议直接优先做。

原因：

- 你已经有动态扫描本地模型
- 当前最强差异不是“再做一个本地模型下载器”
- 更值得继续做的是“模型进来之后怎么被比较、评估、运维”

## A1. 适合移植，而且长进来会很顺的能力

下面这些是我认为最值得纳入现有项目路线的，按优先级排序。

### 1. Studio Recipe Gallery

灵感来源：

- Google AI Studio 的 Prompt Gallery
- LM Studio 的 Presets / import / publish

#### 为什么适合你项目

这会自然长在现在的 Compare Lab、Benchmark preset、prompt template 体系里。

它不是一个新产品，而是把你已经有的：

- compare recipe
- benchmark preset
- prompt template
- output schema
- target set

收成“可复用、可分享、可导入”的工作流配方。

#### 在你项目里的形态

`/agent`
- `Recipe Gallery`
- 场景卡：
  - JSON extraction
  - Tool-first agent run
  - Grounded QA
  - Repo evidence compare
  - Local vs remote latency check

`/admin`
- benchmark starter recipes
- regression recipes

#### 为什么它会很创新

因为你不是在分享“prompt 点子”，而是在分享：

- prompt
- schema
- target matrix
- fairness controls
- benchmark handoff

这比普通 prompt gallery 更像“可执行实验配方”。

### 2. Get Code / Reproduce Request

灵感来源：

- Google AI Studio 的 `Get code`

#### 为什么适合你项目

你项目很强的一点是：当前请求并不只是 prompt，还包括：

- target
- provider profile
- thinking mode
- tool loop
- retrieval
- schema
- compare shape

如果这些能一键导出成：

- Python
- TypeScript
- `curl`

就能把工作台里的实验，直接变成“工程里可复现的调用”。

#### 在你项目里的形态

`/agent`
- chat 请求导出 SDK snippet
- compare lane 导出 SDK snippet

`/admin`
- benchmark recipe 导出 runnable harness snippet

#### 为什么它会很创新

因为导出的不只是“单次聊天代码”，而是：

- 带 output contract
- 带 tool config
- 带 fairness fingerprint
- 带 provider-specific knobs

这会把你的产品从“看结果的面板”推进到“生成工程级复现入口的工作台”。

### 3. Capability Chips / Run Settings Chips

灵感来源：

- Google AI Studio 的 Run settings / AI Chips

#### 为什么适合你项目

你项目现在已有：

- tools
- retrieval
- structured output
- thinking mode
- provider profile

但这些目前更像散落控件。把它们抽成高层 capability chips，会让用户理解更直接。

#### 在你项目里的形态

`/agent`
- 主 composer 顶部显示：
  - `Structured output`
  - `Tool loop`
  - `Grounded / Retrieval`
  - `Thinking`
  - `Strict compare`

`/admin`
- benchmark recipe 中也用同样的能力标签

#### 为什么它不会违和

因为这只是把已有能力“产品化表达”，不是加一套新逻辑。

### 4. Provider Usage Ledger / API Health Desk

灵感来源：

- Google AI Studio 的 usage / key / prototyping console
- LM Studio 的 server settings / auth / local infra 视角

#### 为什么适合你项目

你已经在做：

- runtime telemetry
- provider profile
- benchmark history

下一步很自然就是给远端 provider 增加：

- 最近成功请求
- 最近 429 / timeout / auth failure
- token / request / rough cost summary
- provider-specific fallback history

#### 在你项目里的形态

`/admin`
- 新增 `Provider desk`
- 对每个远端目标展示：
  - status
  - recent failures
  - latency trend
  - token volume
  - estimated cost

#### 为什么它很值当

这会把“远端 API 只是一个 target”升级成“远端 provider 也是可运营对象”。

### 5. Prompt-to-Workflow Drafts

灵感来源：

- Google AI Studio 的 Build mode

#### 为什么适合你项目

不建议把 Build mode 直接翻译成“生成完整 app”，但可以把它翻译成：

> 让用户用自然语言描述自己要做的实验，然后系统给出一个可执行的 compare / benchmark / schema draft。

#### 在你项目里的形态

例如用户输入：

- “我想比较几个模型做 JSON 抽取的稳定性”
- “我想做一个本地 4B 和 GPT-5.4 的 repo 问题对比”

系统自动生成：

- 建议 target set
- output shape
- schema
- fairness settings
- benchmark handoff 草稿

#### 为什么它很创新

这不是 vibe coding，而是：

- `prompt-to-eval`
- `prompt-to-compare`
- `prompt-to-recipe`

非常贴你项目真正的强项。

### 6. Shared Recipe URLs / JSON Import

灵感来源：

- LM Studio preset import / publish

#### 为什么适合你项目

一旦你有 Recipe Gallery，就应该允许：

- 从 JSON 导入
- 从 URL 导入
- 导出 recipe bundle

这样团队协作和社区扩散都更容易。

#### 在你项目里的形态

- compare recipe export
- benchmark recipe export
- prompt template bundle export

### 7. Runner Pool / Preferred Machine Routing（后段探索）

灵感来源：

- LM Studio 的 `LM Link`

#### 为什么暂时只放后段

这个方向很有想象力，但现在不该抢优先级。

未来可以探索：

- 多台 Apple Silicon 机器上的本地网关

这很适合作为 `v0.4+` 的 infra 方向，而不是现在立刻做。

## A. 从 LM Studio 截图里提炼出的功能价值

截图里最值得注意的不是“控件很多”，而是它把三类能力放在同一处：

- 模型加载参数
- 推理采样参数
- 结构化输出 / 提示词模板 / 草稿模型这种更偏高级工作流能力

对我们来说，应该按“是否真的帮助新手微调和输出对比”来筛，而不是照单全收。

### 1. 直接值得加入 v0.3.0 的能力

| 功能 | 在截图中的表现 | 对新手微调的价值 | 对输出对比的价值 | 结论 |
| --- | --- | --- | --- | --- |
| 上下文长度 | `上下文长度` slider/input | 高。会直接影响长样本训练/评测和推理截断行为理解。 | 高。对比模型时必须统一预算。 | 进入 v0.3.0 |
| 采样参数基础版 | `温度`、`Top K`、`Top P`、`Min P`、`重复惩罚` | 中。适合帮助新手理解“微调没变好”与“采样太飘”是两件事。 | 高。输出对比时需要锁定参数。 | 进入 v0.3.0，但要分“基础/高级” |
| 随机种子 | `种子` | 高。复现训练前后输出、减少“偶然答对”误判。 | 高。A/B 对比时很重要。 | 进入 v0.3.0 |
| 限制响应长度 | `限制响应长度` | 中。帮助新手区分“啰嗦”是模型风格还是 `max_tokens` 设置。 | 高。对比时应该统一长度预算。 | 进入 v0.3.0 |
| 停止字符串 | `停止字符串` | 中。对模板微调、JSON 截断、特殊结束符很有帮助。 | 中高。结构化输出和代码对比时很有用。 | 进入 v0.3.0 高级区 |
| 结构化输出 | `结构化输出` | 高。新手最容易从这里看到“模型是否更稳地 obey schema”。 | 很高。对比结果最客观。 | 进入 v0.3.0 |
| 提示词模板 | `提示词模板` | 很高。新手微调经常栽在 chat template 和 instruction formatting。 | 高。不同模板会显著改变输出。 | 进入 v0.3.0 |
| Remember settings / 每模型默认参数 | `Remember settings for ...` | 高。让实验可复用，不用手动记参数。 | 高。方便保存“公平对比 preset”。 | 进入 v0.3.0 |
| 上下文溢出策略 | `上下文溢出` | 中高。对长文任务和训练样本调试很关键。 | 中高。统一策略后对比才公平。 | 进入 v0.3.0 |

### 2. 适合进入高级模式，但不该先摆在首页的能力

| 功能 | 在截图中的表现 | 价值判断 | 结论 |
| --- | --- | --- | --- |
| Reasoning Parsing | `Reasoning Parsing` | 对 reasoning 模型的显示和清洗有价值，但不是新手第一优先级。 | 进入高级区，晚于基础推理参数 |
| 草稿模型 / 投机解码 | `草稿模型` | 更偏吞吐优化，不是新手微调的核心问题。 | 放在 v0.3.x 后段 |
| Max Concurrent Predictions | `Max Concurrent Predictions` | 更偏服务吞吐和队列管理，不是单用户实验第一优先级。 | 先放 Admin Runtime 高级区 |
| KV 缓存量化 | `KV 缓存量化` | 更偏内存优化和性能取舍，适合进阶用户。 | 晚于定时卸载和 idle TTL |

### 3. 不建议一开始就强推给新手的能力

这几项不是没用，而是太容易把产品带成“参数游乐场”：

- 同时开放太多采样旋钮而没有 preset
- 过早暴露吞吐优化参数而没有资源估算
- 把“推理参数调优”和“微调效果提升”混在一起

我们的产品策略应该是：

- 默认只给基础参数
- 用 preset 保存公平对比条件
- 把专家参数折叠到高级面板
- 所有实验都鼓励“锁 seed + 锁模板 + 锁采样 + 锁 context”后再比较

## B. 最值得移植的 4 个功能：产品方案 + IA + 技术清单

之前确认过最值得移植的四项是：

1. Per-target defaults
2. JIT / auto-unload / retention policy
3. MCP 管理 UI
4. Presets / Prompt Library

下面把它们细化成可执行方案。

### B1. Per-target defaults

#### 产品目标

让每个模型目标都能保存自己的默认实验参数，而不是全局共用一套设置。

#### 适用场景

- `Qwen3.5 4B` 默认是 `32K + balanced + standard + structured off`
- `0.6B` 默认是轻量模式
- 某个远端 target 默认是 `speed`
- 某个本地 target 默认绑定一套 prompt template 和输出 schema

#### 页面信息架构

`/agent`
- 目标卡片右上角增加 `Defaults`
- 点击后打开 `Target Defaults Drawer`
- 分区：
  - Runtime defaults
  - Inference defaults
  - Prompt defaults
  - Output defaults

`/admin`
- Local / Remote target cards 增加 `Default profile` 摘要
- Benchmark 配置区可一键使用某目标默认参数

#### 技术实现清单

数据层：
- 在 [types.ts](../lib/agent/types.ts) 增加 `AgentTargetDefaults`
- 在 `lib/agent` 下新增 target defaults store，建议落盘到 `data-dir`

前端：
- 在 [AgentWorkbench.tsx](../components/agent/AgentWorkbench.tsx) 增加 per-target defaults editor
- 在 [AdminDashboard.tsx](../components/admin/AdminDashboard.tsx) 显示摘要与“应用默认值”入口

后端：
- 新增 `/api/agent/targets/defaults` GET/POST
- chat / stream / benchmark 入口在未显式传参时自动读取 target defaults

#### v1 范围

- contextWindow
- maxTokens
- providerProfile
- thinkingMode
- temperature / seed
- stop strings
- prompt template
- structured output schema reference

## C. 新增功能：本地模型“定时从内存清理”

这是目前最值得直接补到产品里的 runtime 功能。

LM Studio 这类工具已经在“Idle TTL / Auto-Evict / Only Keep Last JIT Loaded Model”方向上证明过价值。对你的项目来说，这个能力甚至比继续增加一个 benchmark 图更实用。

### 产品定义

在每个本地模型卡片上允许用户设置：

- 永不自动卸载
- 空闲 5 分钟后卸载
- 空闲 15 分钟后卸载
- 空闲 30 分钟后卸载
- 空闲 60 分钟后卸载
- 自定义时长

并支持：

- benchmark 运行中跳过自动卸载
- 当前会话正在使用该模型时暂停计时
- 切换到别的本地模型后，旧模型按策略自动释放

### 页面信息架构

`/agent`
- 本地运行时卡片新增 `Auto unload` 子区
- 展示：
  - 当前策略
  - 剩余倒计时
  - 上次自动卸载时间
  - 上次自动卸载原因

`/admin`
- Runtime Ops 每个本地目标卡片新增：
  - TTL selector
  - `pause when active` toggle
  - `keep last loaded only` toggle
  - auto-unload event history

### 技术实现清单

后端：
- 扩展 [AgentRuntimeStatus](../lib/agent/types.ts) 增加：
  - `lastUsedAt`
  - `autoUnloadAfterMs`
  - `autoUnloadPaused`
  - `nextAutoUnloadAt`
  - `lastAutoUnloadAt`
  - `lastAutoUnloadReason`
- 扩展 [AgentRuntimeAction](../lib/agent/types.ts) 增加：
  - `set_auto_unload_policy`
- 在 [runtime-state.ts](../lib/agent/runtime-state.ts) 记录 target last-used
- 在 `lib/agent` 新增 runtime retention store
- 增加周期性 scheduler，触发 `/v1/models/release`

前端：
- [AgentWorkbench.tsx](../components/agent/AgentWorkbench.tsx)
- [AdminDashboard.tsx](../components/admin/AdminDashboard.tsx)

### 验证标准

- 空闲超时后模型自动释放
- 释放时不卡死当前页面
- benchmark 中不会误卸载
- 切模型后旧模型能按策略释放
- UI 能解释“为什么没卸载 / 为什么刚刚卸载”

## D. 为“模型输出对比”新增的能力

如果我们想在**现有 Agent 工作台里**补出更强的模型实验能力，应该增加下面这些对比能力。

### D1. Inference Compare Lab

#### 产品目标

让用户在同一个 prompt 下，对多个模型、多个 preset、多个 schema 输出做可重复的横向对比。

详细规格：

- [`docs/inference-compare-lab-spec.md`](./inference-compare-lab-spec.md)

#### 页面信息架构

`/agent` 内的 compare 模式（v1）；仅当使用密度明显提升后，才考虑独立成 `/agent/compare` 子路由。
- 左侧：
  - prompt
  - prompt template
  - inference preset
  - output schema
- 中间：
  - target matrix
- 右侧：
  - diff view
  - schema pass/fail
  - token / latency / stop reason

#### 首批支持的比较维度

- 相同 prompt，不同模型
- 相同模型，不同采样 preset
- 相同模型，不同 prompt template
- 微调前 vs 微调后 checkpoint / adapter
- 结构化输出成功率
- 长上下文截断行为

### D2. Fair comparison guardrails

为了避免“看起来在对比，其实口径不一致”，需要系统强制记录：

- contextWindow
- maxTokens
- seed
- temperature / top-k / top-p / min-p / repetition penalty
- prompt template id
- stop strings
- structured output schema version

## E. 如果后续支持本地模型微调，最适合做哪些功能

这里建议非常明确：

**第一版只做 Apple Silicon 上可落地的 LoRA / QLoRA / adapter 微调工作流。**

不要一开始就做：

- 全参微调工作台
- RLHF / DPO / PPO 全家桶
- 多机分布式训练编排
- 云训练平台替代品

### E1. 适合 v0.3.x 第一阶段的微调功能

#### 1. Dataset import + validation

- 导入 JSONL / CSV / parquet / HF dataset
- 识别常见字段：instruction / input / output / system / messages
- 抽样预览
- 检查：
  - 空样本
  - 重复样本
  - 超长样本
  - 模板不匹配样本

#### 2. Prompt template / chat template mapper

这是新手最容易踩坑的地方，必须做成显式步骤。

- 选择 base model
- 选择 chat template
- 把数据映射到模板
- 预览 tokenized prompt
- 显示会被截断的位置

#### 3. Fine-tune recipe presets

第一版只提供少量明确 preset：

- `Quick LoRA smoke`
- `Balanced LoRA`
- `Long-context SFT check`
- `Classification / extraction`
- `JSON obedience`

每个 preset 固定住：

- learning rate
- batch size
- gradient accumulation
- sequence length
- LoRA rank / alpha / dropout
- target modules

#### 4. Job queue + runtime monitor

- 训练任务排队
- 当前 epoch / step
- loss 曲线
- tokens/sec
- 内存占用
- 预计剩余时间
- 可暂停 / 取消

#### 5. Checkpoint registry

- base model
- adapter name
- dataset version
- recipe version
- metrics
- notes
- createdAt

#### 6. Before/after compare + benchmark

这是你项目最有优势的地方，必须比“只会训练”更强：

- base vs adapter compare
- fixed prompt sets
- structured output compare
- repo question compare
- milestone-formal compare

#### 7. One-click deploy to local runtime

- 训练完成后把 adapter 注册成新 target
- 在 `/agent` 里直接可选
- 在 `/admin` benchmark 里直接可跑

### E2. 页面信息架构

第一阶段不建议把它做成“另一个新产品”。更合适的方式是：

- 先作为 `/admin` 下的实验能力，或 feature flag 子页面接入
- 复用现有 app shell、导航、runtime 与 benchmark 基础设施
- 等微调链路真实跑顺以后，再评估是否需要独立成稳定子路由

建议的实验信息架构：

#### `/admin` 下的 finetune flow（实验）

分成 5 个步骤：

1. Dataset
2. Template
3. Recipe
4. Train
5. Compare & Deploy

#### `/admin`

保留：

- fine-tune job summary
- recent runs
- adapter registry shortcut

### E3. 技术实现清单

训练后端建议：
- 优先对接 `mlx-lm` 的 LoRA / quantized fine-tuning 能力
- 只支持 Apple Silicon 本地训练
- 训练任务单独 worker 进程
- 结果落盘到独立 `finetune` data dir

建议新增目录：
- `lib/finetune/`
- `app/api/finetune/`
- `components/finetune/`
- `scripts/finetune_*`

### E4. v1 不做的事

- 自动帮用户挑最优超参
- 多节点训练
- RLHF / preference optimization 全流程
- 训练数据自动抓取器
- 云端训练 marketplace

## F. 建议的版本推进顺序

### Phase 1: v0.3.0 alpha

先做最影响日常体验的三件事：

1. Per-target defaults
2. 定时卸载 / idle TTL / retention policy
3. Inference Compare Lab v1

### Phase 2: v0.3.0 beta

4. Prompt templates
5. Structured output compare
6. Presets / prompt library
7. 上下文溢出策略
8. seed / stop strings / sampling advanced panel

### Phase 3: v0.3.x

9. MCP 管理 UI
10. reasoning parsing
11. speculative decoding / draft model
12. 并发预测参数
13. KV cache quantization

### Phase 4: v0.4.0 candidate

14. Fine-tune Dataset + Template + Recipe
15. Training jobs + monitor
16. Adapter registry
17. Before/after benchmark lane
18. One-click deploy to local runtime

## G. 结论

如果按“投入产出比”排序，下一阶段最值当的不是再加更多 benchmark 图，也不是另起一套新产品，而是在原项目里先把下面四条做成真正的工作流：

1. 每模型默认参数
2. 定时卸载 / 空闲自动释放
3. 可重复的输出对比实验台
4. 面向 Apple Silicon 的 LoRA 微调与前后对照闭环

这四条一旦做出来，Local Agent Lab 就会从“很强的 benchmark + runtime ops 面板”，进一步升级成“真正可用的本地模型实验工作台”。

## H. 把这次竞品对比真正落进版本规划

基于 Google AI Studio 和 LM Studio 的对比，建议把新增方向收成下面这个版本序列。

### Phase 1.5: v0.3.0 beta+

优先补两条最有产品感、又不违和的能力：

1. `Studio Recipe Gallery`
2. `Capability Chips / Run Settings Chips`

原因：

- 它们能最快提升“第一眼理解度”
- 不需要推翻现有 `/agent` 与 `/admin`
- 对 compare / benchmark 两条主线都有效

### Phase 2: v0.3.x

在现有计划基础上，把下面两条提前：

3. `Get Code / Reproduce Request`
4. `Shared Recipe URLs / JSON Import`

原因：

- 这两条最能把“实验台”推进到“工程工作台”
- 也是最适合 GitHub / 社区传播的亮点

### Phase 3: v0.3.x late

把运维感更强的能力接上：

5. `Provider Usage Ledger / API Health Desk`
6. `MCP 管理 UI`

### Phase 4: v0.4+ exploratory

7. `Prompt-to-Workflow Drafts`
8. `Runner Pool / Preferred Machine Routing`

这两条都很有想象力，但需要建立在：

- recipe system 已经成型
- provider desk 已经稳定
- runtime / benchmark 基础设施足够稳

之上。
